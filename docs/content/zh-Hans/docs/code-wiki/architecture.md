# 架构设计

## 3.1 整体架构

Project Airi 采用分层架构设计，从下至上依次为基础设施层、数据层、业务逻辑层、接口层和表现层。

### 基础设施层

包含数据库服务、缓存服务和文件存储服务，为上层应用提供稳定的数据存储和访问能力。

### 数据层

通过 Drizzle ORM 定义数据模型和数据库操作接口，实现数据访问的抽象和封装。

### 业务逻辑层

包含各种服务类，处理具体的业务规则和数据流转逻辑。

### 接口层

提供 RESTful API 和 WebSocket 接口，供前端和客户端应用调用。

### 表现层

则是面向用户的各种应用界面，包括 Web 页面、桌面应用和移动应用。

## 3.2 Monorepo 架构

项目使用 pnpm workspace 管理 monorepo 结构。这种架构模式带来了代码复用率高、版本管理统一、跨项目重构方便等优势。

```yaml
# pnpm-workspace.yaml
packages:
  - 'apps/*'
  - 'packages/*'
  - 'services/*'
```

各个应用和包之间通过 workspace 协议引用，无需手动管理 node_modules。

## 3.3 前端状态管理

前端采用 Pinia 进行状态管理，定义了多个 Store 模块：

- **auth Store**：管理用户认证状态，包括登录状态、用户信息和访问令牌
- **chat Store**：管理聊天相关的状态，包括聊天记录、当前对话和消息发送状态
- **character Store**：管理角色相关的数据，包括角色列表、角色详情和角色配置

## 3.4 后端分层架构

后端采用经典的三层架构模式：路由层（Routes）、服务层（Services）和数据访问层（Schemas）。

### 路由层

负责接收 HTTP 请求，进行参数验证和初步处理，然后调用服务层完成业务逻辑。

### 服务层

封装具体的业务规则，处理复杂的数据操作和业务决策。

### 数据访问层

定义数据库表结构，提供基础的 CRUD 操作接口。

### 中间件层

提供横切关注点的处理能力：
- **auth.ts**：JWT 令牌验证中间件
- **cors.ts**：跨域资源共享配置
- **error.ts**：全局错误处理中间件

### 工具层

提供各类辅助函数，如 ID 生成、日期格式化、加密解密等。

## 3.6 数据流

1. 用户交互触发 API 请求
2. 前端发送带有认证令牌的请求
3. 后端中间件验证认证
4. 路由处理器验证参数
5. 服务层执行业务逻辑
6. 数据访问层查询/更新数据库
7. 响应通过各层返回前端
